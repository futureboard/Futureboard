# SKILL.md — Futureboard Studio / Sphere Agent Skill

## Purpose

This skill teaches Claude Code, Codex, and other coding agents how to work safely inside the Futureboard Studio / Mochi DAW codebase.

Futureboard Studio is a DAW built around a modern hybrid stack:

- React / Electron for the current product UI
- Rust for safe native integration, audio infrastructure, and N-API modules
- C++ for low-level native SDK integration where required
- WebAudio / WASM as web-compatible fallback paths
- Native services for long-term realtime audio, plugin hosting, and heavy processing

This skill is for controlled implementation.

It is not permission to rewrite the entire repository.

---

## When To Use This Skill

Use this skill when the user asks to:

- continue Futureboard Studio / Mochi DAW development
- implement a task from `tasks/`
- work from a numbered task file such as `006-DawSystem.txt`
- fix a DAW UI, state, timeline, mixer, audio, plugin, window, or platform bug
- wire menu, state, project, transport, routing, audio, plugin, or window systems
- implement an incremental patch from a long task spec
- work on `SpherePluginHost`
- work on `PluginHost.node`
- work with `external/vst3sdk`
- build VST3 / CLAP / AU plugin host infrastructure
- bridge native systems into Electron
- create safe Rust/C++ native module boundaries

## 0. Prime Directive

Always work from the smallest safe scope.

Before changing code:

1. Read the relevant user request.
2. Read only the relevant task file or section.
3. Inspect the current implementation.
4. Identify the smallest safe patch.
5. List the likely files to change.
6. Implement only the requested scope.
7. Run the relevant build/check command.
8. Report what changed, what was validated, and what remains.

Never implement an entire long task file unless the user explicitly requests it.

Never rewrite the whole app unless the user explicitly requests a rewrite.

## 4. Standard Implementation Pattern

When given a task, follow this pattern:

1. Restate the exact task scope.
2. Read the relevant task file/section.
3. Inspect existing files.
4. List files likely to change.
5. Implement the smallest safe patch.
6. Run build/check.
7. Summarize:
   - changed files
   - behavior added/fixed
   - validation result
   - intentional TODOs

Example user instruction:

```txt
Read tasks/webapp/006-DawSystem.txt.
Implement only section 3 and 4.
```

Correct behavior:

- Implement only Time/Musical Math and Timeline Coordinate System.
- Do not implement the whole WebAudio engine.
- Do not rewrite the UI.
- Build and stop.

Incorrect behavior:

- Implement all of `006-DawSystem.txt`.
- Rewrite project store, audio engine, UI, commands, and theme in one patch.
- Change unrelated styles.
- Add random dependencies.

Do:

- Make small patches.
- Preserve existing working behavior.
- Use existing stores/actions/selectors where possible.
- Centralize duplicated logic.
- Add safe placeholders for future-heavy features.
- Disable unimplemented UI actions.

- Keep Rust warnings low.
- Keep C++ boundaries explicit.
- Run build/check after implementation.
- Keep Web/Electron/native boundaries clean.
- Use platform adapters instead of direct Electron/Node calls in React.
- Treat current React/Electron app as the living spec.

Do not:

- Rewrite the entire app unless explicitly requested.
- Rewrite audio engine when doing a UI task.
- Implement native DSP when doing a UI task.
- Implement VST/CLAP/AU hosting unless specifically requested.
- Add random dependencies.
- Add fake-success actions.
- Duplicate project state in multiple stores.
- Put heavy audio buffers into React persisted state.
- Parse JSON in realtime audio paths.
- Allocate heavily in realtime audio paths.
- Log from realtime audio callbacks.
- Block realtime threads.
- Break mouse/keyboard workflow while adding touch support.
- Replace the design system without permission.
- Move all processing into WebUI.
- Route realtime audio through Node/Electron.

## 7. TypeScript Rules

Avoid syntax that may break strict or erasable TypeScript settings.

Avoid:

- TypeScript parameter properties if the repo rejects them
- huge `any` usage
- unchecked optional fields
- mutating state directly
- duplicate types in random files
- direct Electron imports inside React UI
- direct `fs` or `child_process` usage inside React UI

Prefer:

- explicit interfaces/types
- narrow type guards
- normalized defaults
- selectors
- immutable updates
- small helpers
- adapter interfaces
- stable event payloads
- debounced/throttled high-frequency UI updates

---

## 8. Rust Rules

Rust is used for safe native boundaries, host state, N-API modules, audio infrastructure, and future service glue.

Do:

- keep unsafe code isolated
- document every unsafe FFI assumption
- wrap raw pointers in explicit handles
- use `Result<T, E>` across fallible APIs
- use `thiserror` or project-native error patterns
- avoid panics across FFI
- avoid blocking inside audio callbacks
- avoid allocations on realtime paths
- keep N-API functions thin
- move heavy native logic outside JS-visible wrappers

Do not:

- expose raw C++ pointers directly to JS
- let panics cross FFI
- use global mutable state casually
- hold locks in realtime audio paths
- perform filesystem scanning on the audio thread
- serialize large buffers through JSON
- use Node as part of realtime processing

---

## 9. C++ Rules

C++ is allowed where required for native SDK integration.

Current likely use cases:

- VST3SDK integration
- plugin scanning/loading
- plugin editor view handling
- native window handles
- ABI bridge for Rust
- low-level OS-specific audio/plugin APIs

Do:

- keep C++ behind a small C ABI or explicit bridge layer
- isolate VST3SDK headers from Rust
- avoid leaking SDK types into N-API
- own plugin lifecycle clearly
- make destroy/free functions explicit
- handle plugin crashes defensively where possible
- keep platform-specific window code separated

Do not:

- spread VST3SDK types through the whole app
- expose C++ exceptions across C ABI
- expose plugin pointers to JavaScript
- allocate/log/block in realtime process functions
- make React depend on C++ headers
- merge UI window code into audio processing code

## 10. SpherePluginHost Rules

`SpherePluginHost` is the native plugin host layer.

Preferred direction:

```txt
Electron / React
  -> preload IPC / native API
    -> PluginHost.node
      -> Rust N-API wrapper
        -> C ABI
          -> C++ VST3 Host Core
            -> external/vst3sdk
              -> VST3 plugins
```

Allowed:

- `external/vst3sdk`
- Rust wrapper
- C++ VST3 host core
- N-API module output as `PluginHost.node`
- later separate process: `FutureBoardPluginHost.exe`

Do not use JUCE unless the user explicitly asks.

Do not force JUCE into the project indirectly.

Do not build the first version as a giant universal host.

Start with VST3 scanner and metadata.

---

## 11. PluginHost.node Rules

`PluginHost.node` is a control bridge, not the realtime audio engine.

Good uses:

- initialize plugin host
- scan plugin folders
- list discovered plugins
- load/unload plugin instances
- query parameters
- set parameter values
- save/load plugin state
- open/close native plugin editor windows
- report plugin metadata
- communicate with a separate plugin host process later

Bad uses:

- realtime audio processing through Node
- sending audio buffers through JSON
- large waveform/audio transfer
- blocking filesystem scan during playback
- loading untrusted plugins directly in the UI process without a future isolation plan

Preferred first API surface:

```ts
initPluginHost(): void
shutdownPluginHost(): void

scanVst3(paths: string[]): Promise<PluginInfo[]>

loadPlugin(options: LoadPluginOptions): string
unloadPlugin(instanceId: string): void

getParameters(instanceId: string): ParameterInfo[]
setParameter(instanceId: string, parameterId: number, normalizedValue: number): void

saveState(instanceId: string): Uint8Array
loadState(instanceId: string, state: Uint8Array): void

openEditor(instanceId: string, parentWindowHandle?: bigint): void
closeEditor(instanceId: string): void
```

The first implementation may return JSON strings internally between C++ and Rust if that keeps the bridge simple.

Do not keep JSON in realtime paths.

## 12. VST3 Host Rules

VST3 support may use `external/vst3sdk`.

That does not mean the whole app is a VST3SDK app.

Keep VST3SDK usage contained.

Recommended phases:

### Phase 1 — Scanner

Implement:

```txt
scanVst3(paths) -> PluginInfo[]
```

Goal:

- discover `.vst3`
- read factory metadata
- return name/vendor/category/class id/path
- cache scan results if existing project patterns support it

Do not:

- process audio
- open editor
- build full graph
- implement automation

### Phase 2 — Load/Unload

Implement:

```txt
loadPlugin(path/classId)
unloadPlugin(instanceId)
```

Goal:

- instantiate component/controller
- initialize basic lifecycle
- return stable instance handle

Do not:

- route realtime audio through JS
- expose raw pointers

### Phase 3 — Parameters

Implement:

```txt
getParameters(instanceId)
setParameter(instanceId, paramId, normalizedValue)
```

Goal:

- query parameter metadata
- set normalized values
- sync to UI/store
- prepare automation model later

### Phase 4 — State

Implement:

```txt
saveState(instanceId)
loadState(instanceId, state)
```

Goal:

- project save/load support
- preset-like binary state handling

Do not:

- store state as huge JSON
- mutate unrelated project model

### Phase 5 — Audio Process

Implement native-only processing path.

Preferred path:

```txt
SphereDirectAudioEngine
  -> native plugin graph
    -> SpherePluginHost C++ core
      -> VST3 process
```

Do not:

```txt
AudioEngine -> Node -> PluginHost.node -> VST3
```

### Phase 6 — Editor Window

Implement native editor windows carefully.

Preferred first approach:

- external native plugin editor window
- not embedded inside React
- safe focus/resize handling
- platform-specific window code isolated

Embedding into Electron can come later.

## 13. Plugin Crash Isolation

A `.node` module runs in the Electron process.

If the plugin crashes, the app may crash.

Short-term:

- acceptable for scanner/load experiments
- keep operations minimal
- fail safely when possible

Long-term preferred architecture:

```txt
Electron Renderer
  -> Electron Main / Preload
    -> PluginHost.node client
      -> IPC
        -> FutureBoardPluginHost.exe
          -> VST3/CLAP/AU plugins
```

The separate process should eventually provide:

- crash isolation
- plugin sandboxing where possible
- recovery after plugin failure
- safer scanning
- editor process isolation if needed

Do not claim plugin hosting is production-safe until isolation exists.

## 14. Realtime Audio Rules

Realtime audio paths must avoid:

- allocations
- locks
- blocking IO
- filesystem access
- JSON parsing
- logging
- panics/exceptions
- Node/Electron calls
- IPC roundtrips
- dynamic plugin scanning
- unbounded queues
- waiting on UI thread

Allowed in realtime paths:

- preallocated buffers
- lock-free queues if already designed safely
- atomic parameter values
- fixed-size event queues
- native DSP calls
- native plugin process calls after proper setup

React/Electron can control audio.

React/Electron must not become the audio path.

- Do not send huge binary audio data over JSON.

## 17. State Rules

Single source of truth for project data:

- project store / project state

Important ownership:

- tracks live in project state
- clips live in track/project state
- MIDI notes live in MIDI clip state
- mixer values live in track mixer state
- inserts live in `track.inserts`
- device params live in insert device params
- routing lives in track routing/sends
- plugin instances must map cleanly to insert IDs
- plugin scan registry is not the same as project state
- transient meters should not cause full project rerenders
- transport high-frequency movement should be isolated/throttled

Never:

- create local-only fake state for persistent DAW data
- create a second insert chain outside `track.inserts`
- create a second mixer model outside track mixer state
- create a disconnected plugin gallery as the real plugin system
- persist huge audio buffers in React state

## 18. Platform Boundary Rules

Web:

- no Node APIs
- safe fallbacks
- browser storage via IndexedDB/OPFS where appropriate
- WebAudio/WASM fallback only

Electron:

- renderer uses preload IPC
- main handles filesystem/dialog/process
- no direct `fs`, `child_process`, or Electron imports inside React UI
- native modules are loaded only in approved boundary layers

Native service:

- spawned by Electron main
- renderer talks through preload
- service crash must not crash UI
- fallback to WebAudio/WASM if unavailable
- do not make service mandatory until stable

## 23. Dependency Rules

Do not add dependencies casually.

Before adding a dependency, check:

- Is there already a project utility for this?
- Does the dependency work in Web and Electron?
- Does it break packaging?
- Does it add native build complexity?
- Is it compatible with Bun/Vite/Electron?
- Is it safe for realtime or native boundaries?
- Does it pull in JUCE or other unwanted frameworks indirectly?

For plugin host work:

- VST3SDK under `external/vst3sdk` is acceptable when requested.
- JUCE is not acceptable unless explicitly requested.
- CLAP SDK may be acceptable only when explicitly requested.
- Audio plugin hosting must remain modular by format.

## 24. Scanner / Registry Rules

Plugin scanner and plugin registry are separate from loaded plugin instances.

Scanner:

- discovers plugins
- reads metadata
- can cache results
- must survive bad plugins where possible
- should support rescanning
- should avoid blocking the UI

Registry:

- stores known plugin metadata
- may be persisted outside project state
- can be used by plugin selector UI
- should not store live instance pointers

Project inserts:

- store selected plugin/device identity
- store plugin state
- store parameter values/automation
- map to live instances only when project is loaded

2. no React embedding first
3. no audio processing changes
