Task: Make Insert Plugins Actually Process Audio, Then Upgrade Plugin UI to Production DAW Quality

Context:
The current plugin system can add insert devices into `track.inserts`, and the Effect Editor can render plugin editors. However, the audio engine does not currently route track audio through those insert devices.

Observed current behavior:
- User clicks Add Insert / Add Equz8 / Add UltraVerb / Add UltraDelay / Add FB-2A Comp.
- Device appears in Mixer and/or Effect Editor.
- Parameter UI changes are stored in project state.
- Playback audio does not change.

Likely root cause from current code:
- `ClipScheduler` connects each playing clip into `mixer.getOrCreateTrack(track.id, track.volume, track.pan).gain`.
- `Mixer` currently builds a simple fixed track graph:
  `track gain -> stereo panner -> analyser split -> track meter -> master`
- There is no insert DSP graph between track input and track output.
- `track.inserts` is not synchronized into WebAudio nodes.
- Plugin definitions have UI and parameter schemas, but no DSP factory/runtime binding.

Goal:
Implement a real, working built-in insert DSP runtime for the current WebAudio prototype, and improve the plugin editors so they feel like professional DAW devices instead of rough mockups.

The first working target is:
- Add an insert.
- Play audio.
- Hear the insert affect the sound immediately.
- Change plugin parameters while playing.
- Hear smooth realtime changes without stopping playback.
- Bypass/remove/reorder inserts without breaking the audio graph.
- UI looks compact, horizontal, polished, and production-grade.

This task is not VST/CLAP hosting.
This task is not the full native/Rust rewrite.
This task must keep the current WebAudio prototype working.

0. Non-Negotiable Rules
============================================================

Do:
- Keep `track.inserts` as the single source of truth.
- Build the runtime around the existing `Mixer`, `ClipScheduler`, and plugin registry.
- Make WebAudio DSP work first.
- Use AudioWorklet only where native WebAudio nodes are not enough or where latency/quality requires it.
- Smooth parameter changes using `setTargetAtTime`, `linearRampToValueAtTime`, or equivalent safe ramps.
- Keep parameter updates realtime-safe: no heavy graph rebuild on every slider tick unless unavoidable.
- Keep UI compact and horizontal.
- Run build after changes.
- Manually test playback with at least EQ, delay, reverb, and compressor.

Do not:
- Create a second fake insert chain.
- Keep plugin DSP disconnected from `track.inserts`.
- Put plugin audio state in React component local state as the source of truth.
- Allocate, parse JSON, log repeatedly, fetch, or block inside realtime audio code.
- Rebuild convolver impulse responses on every mousemove.
- Add random dependencies unless there is a strong reason.
- Rewrite the whole DAW.
- Break current clip playback, pitch/speed processing, mixer volume/pan/mute/solo, or metering.

Runtime interfaces:

```ts
export type InsertAudioNode = {
  id: string;
  input: AudioNode;
  output: AudioNode;
  update(params: Record<string, number | string | boolean>, context: InsertUpdateContext): void;
  setEnabled(enabled: boolean): void;
  dispose(): void;
};

export type InsertUpdateContext = {
  now: number;
  sampleRate: number;
  bpm: number;
};

export type InsertNodeFactory = (
  ctx: AudioContext,
  device: InsertDevice,
  context: InsertUpdateContext
) => InsertAudioNode;
```

Runtime responsibilities:
- Create DSP nodes when an insert is first seen.
- Reuse existing DSP nodes when only params change.
- Rebuild the track chain when insert order/add/remove changes.
- Update parameters without rebuilding topology where possible.
- Dispose nodes when insert is removed or track is removed.
- Expose simple methods for `Mixer`:
  - `syncTrackInserts(trackId, inserts, options)`
  - `setTrackTempoContext(bpm)`
  - `removeTrack(trackId)`

Acceptance:
[ ] Runtime can create node instances from plugin type/name/id.
[ ] Runtime reuses nodes across parameter changes.
[ ] Runtime removes stale nodes.
[ ] Runtime can reconnect a chain in order.
[ ] Runtime is independent from React.

5. Built-In DSP: Equz8
============================================================

Implement a real WebAudio EQ for `Equz8`.

Use native `BiquadFilterNode` chain:
- `highpass` -> `BiquadFilterNode.type = "highpass"`
- `lowshelf` -> `"lowshelf"`
- `bell` -> `"peaking"`
- `highshelf` -> `"highshelf"`
- `lowpass` -> `"lowpass"`

For each enabled band:
- frequency Hz -> `filter.frequency`
- gain dB -> `filter.gain` for shelf/peaking
- Q -> `filter.Q` where applicable

For disabled bands:
- Best first pass: keep node in chain but set neutral behavior.
  - peaking: gain 0 dB
  - shelf: gain 0 dB
  - highpass/lowpass: if disabled, bypass node in topology or set harmless extremes.
- Better pass: rebuild only band topology when band enable/type changes.

Output:
- Add output gain if `outputDb`/equivalent exists.
- If current Equz8 params do not include output, add it carefully with default 0 dB.

Analyzer:
- Optional visual analyzer can be UI-only first.
- Later use `AnalyserNode` or shared FFT analyzer tap.

Parameter smoothing:
- frequency/Q/gain/output should ramp over 5-20 ms.
- Avoid zipper noise on slider movement.

Acceptance:
[ ] Boost/cut EQ bands clearly changes tone.
[ ] Highpass removes bass.
[ ] Lowpass removes highs.
[ ] Bypass restores dry tone.
[ ] Changing bands while playing does not click heavily.
[ ] UI curve roughly matches audible behavior.

6. Built-In DSP: UltraDelay
============================================================

Implement a real stereo delay.

First WebAudio graph target:

Dry path:
`input -> dryGain -> output`

Wet path:
`input -> split L/R -> delayL/delayR -> feedback filters/saturation -> feedback loop -> wetGain -> stereo merge -> output`

Minimum viable version:
- Use two `DelayNode`s.
- Use `GainNode` for feedback.
- Use `BiquadFilterNode` highpass/lowpass in feedback path.
- Use dry/wet gains based on `mix`.
- Support free time in ms and tempo sync division via BPM.
- Support ping-pong by cross-feeding L into R and R into L.

Do not exceed safe feedback:
- Clamp feedback to <= 0.98.
- If saturation is not implemented yet, keep it neutral.
- Freeze can hold high feedback later; first pass may map freeze to feedback 0.98 and wet-only behavior if safe.

Tempo sync:
- Use `divisionToMs()` from `plugins/UltraDelay/Core`.
- Need BPM from project/transport context.
- Smooth delay time changes carefully. Delay time changes can pitch-glide; acceptable, but no discontinuous clicks.

Mix:
- Use equal-power dry/wet if possible:
  - dry = cos(mix * pi/2)
  - wet = sin(mix * pi/2)
- Or linear first pass if simpler.

Acceptance:
[ ] Delay is clearly audible at default preset.
[ ] Feedback repeats are audible and stable.
[ ] Ping-pong changes stereo motion.
[ ] LowCut/HighCut darkens or thins repeats.
[ ] Sync divisions follow BPM.
[ ] Bypass passes dry unchanged.
[ ] Feedback never self-explodes at normal values.

7. Built-In DSP: UltraVerb
============================================================

Implement a real reverb that is useful now and replaceable later.

Recommended first pass:
- Use `ConvolverNode` with a generated impulse response.
- Generate IR when major reverb params change:
  - mode
  - size
  - decay
  - damping
  - width
- Do not regenerate IR on every tiny slider movement while dragging.
- Debounce or quantize IR rebuilds to avoid UI/audio stalls.
- Use native filters and gains for realtime controls:
  - preDelayMs -> `DelayNode.delayTime`
  - lowCutHz -> `BiquadFilterNode("highpass")`
  - highCutHz -> `BiquadFilterNode("lowpass")`
  - mix -> dry/wet gains
  - outputDb -> output gain

Suggested graph:

Dry:
`input -> dryGain -> output`

Wet:
`input -> preDelay -> lowCut -> highCut -> convolver -> width stage -> wetGain -> output`

IR generation:
- Length based on decay seconds and sample rate, with reasonable cap.
- Use exponential decay noise.
- Damping can reduce high-frequency content over time.
- Width can generate decorrelated L/R IR channels.
- Mode can change early reflection density/decay shape.

Performance:
- IR generation can allocate, but only outside realtime audio rendering.
- Use debounce around 80-150 ms for sliders that require IR rebuild.
- Never rebuild IR inside an AudioWorklet process callback.

Optional better later:
- Replace Convolver with AudioWorklet Schroeder/FDN reverb.
- Add freeze as a Worklet feature.

Acceptance:
[ ] Reverb is audible at default preset.
[ ] Decay changes tail length.
[ ] PreDelay separates dry sound from reverb onset.
[ ] LowCut/HighCut shape wet signal.
[ ] Mix works.
[ ] Bypass works.
[ ] Moving decay/size does not freeze the app badly.

8. Built-In DSP: FB-2A Comp
============================================================

Implement a real compressor first, then improve opto character later.

Minimum viable WebAudio version:
- Use `DynamicsCompressorNode` for gain reduction.
- Map FB-2A params to compressor params:
  - peakReduction -> threshold lower as value increases
  - mode compress/limit -> ratio/knee behavior
  - emphasis -> sidechain/brightness behavior if possible, or tone coloration later
  - gainDb -> post gain
  - outputTrimDb -> final trim
  - mix -> dry/wet parallel blend
  - stereoLink -> not supported by native node directly; keep for later or implement with Worklet

Suggested graph:

Dry:
`input -> dryGain -> output`

Compressed:
`input -> compressor -> color/tone stage -> makeupGain -> wetGain -> output`

Parameter mapping:
- `thresholdDb = peakReductionToThresholdDb(peakReduction)`
- ratio:
  - compress: 3:1 to 4:1
  - limit: 10:1 to 20:1
- attack: opto-ish slower than FET
  - 5-15 ms
- release:
  - 250-800 ms
- knee:
  - soft, 6-12 dB

Color:
- First pass can be a subtle waveshaper only if easy.
- Noise should default off and must not add loud hiss.
- Avoid hardware clone branding.

Metering:
- Native `DynamicsCompressorNode.reduction` can be sampled for GR display.
- Do not store high-frequency meter data in project state.
- UI should read meter through a transient store/hook or plugin runtime meter API.

Acceptance:
[ ] PeakReduction audibly compresses louder material.
[ ] Gain/output controls change level.
[ ] Mix blends dry/compressed signal.
[ ] Limit mode is stronger than compress mode.
[ ] Bypass works.
[ ] Meter displays approximate GR if implemented.

11. Bypass Design
============================================================

Bypass must sound clean.

Simplest safe bypass:
- Each plugin has dry bypass path and wet/processed path.
- `setEnabled(false)` crossfades:
  - processed gain to 0
  - bypass gain to 1
- `setEnabled(true)` crossfades back.

If plugin graph is naturally inline-only:
- Use a wrapper:
  - `input -> dryBypassGain -> output`
  - `input -> processorInput -> processorOutput -> wetEnabledGain -> output`

Rules:
- Do not disconnect/reconnect on every bypass click if crossfade can handle it.
- Use 5-20 ms ramp.
- UI bypass and audio bypass must use the same `insert.enabled`.

Acceptance:
[ ] Bypass is audible.
[ ] Bypass does not click badly.
[ ] UI enabled state matches audio state.
